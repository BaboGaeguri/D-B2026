# 논문 계획 (스마트팩토리 구매 적용안)

## 하네스 엔지니어링 기반 저비용 LLM의 산업용 로봇 부품 조달 검증 자동화

---

## 한 줄 요약 (쉬운 설명)

산업용 로봇 제조사(FANUC, ABB, KUKA)의 공식 카탈로그를 정답 스펙으로 삼고, 가상 견적서에 의도적으로 오류를 심은 뒤 LLM+하네스가 이를 얼마나 정확히 잡아내는지 측정한다. 정답을 우리가 직접 설계했기 때문에 F1-score로 객관적 평가가 가능하다.

### 실제 적용 흐름

```
산업용 로봇 부품 요청 (예: FANUC 서보모터 A06B-0127)
        ↓
공급업체들이 각자 다른 양식으로 견적서 제출
        ↓
[기존] 구매팀이 카탈로그와 수동 비교 → 스펙 불일치 판단
        ↓
[제안] LLM이 견적서 읽고 카탈로그 대비 적합성 판단
        ↓
[하네스 개입]
  ✓ 제조사 공식 카탈로그 스펙과 일치하는가?
  ✓ 예산 상한선 안에 드는가?
  ✓ 인증 요건을 충족하는가?
  → 실패 항목 발생 시 LLM 재검토 지시
        ↓
구매 담당자에게 검증 완료된 최종 추천 전달
```

> 카탈로그 스펙이 "정답"이기 때문에 하네스의 판단이 맞는지 틀린지
> F1-score로 완전히 객관적으로 측정 가능하다.

---

## 핵심 연구 질문 (Research Question)

> "산업용 로봇 부품 조달 프로세스에서,
> GPT-3.0급 저성능 LLM에 하네스 엔지니어링을 결합하면
> 고성능 모델과 동등한 오류 감지 정확도를 더 낮은 비용으로 달성할 수 있는가?"

"산업용 로봇 부품 조달의 지능화: 하네스 엔지니어링 기반 저비용 LLM의 스펙 검증 정확도-비용 최적화 설계"

## 왜 이 접근인가

### 기존 계획의 한계

기존 논문 계획은 나라장터 공공 조달 데이터를 사용했으나, 이 데이터에는 **"이 견적이 맞다/틀리다"를 판단할 정답이 없었다.** LLM 출력의 품질을 객관적으로 측정할 수 없다는 근본적 문제가 있었다.

```
기존 방식의 문제
├─ 나라장터 데이터 → 낙찰 결과만 있음, 스펙 불일치 여부 라벨 없음
├─ 실제 기업 견적서 → 기업 기밀, 접근 불가
└─ → LLM이 잘 판단했는지 측정 불가
```

### 해결: 제조사 카탈로그 = 공개된 정답

산업용 로봇 주요 제조사들은 부품 스펙을 공식 카탈로그로 공개한다.
이것이 "정답 스펙" 역할을 하며, 견적서와의 불일치를 객관적으로 정의할 수 있다.

| 제조사 | 공개 데이터 | 활용 방법 |
| --- | --- | --- |
| FANUC | 서보모터·드라이버 카탈로그 | 부품번호별 정격 토크·전압·전류 정답 |
| ABB | 로봇 부품 사양서 | 감속기·베어링 허용 하중 정답 |
| KUKA | 부품 교체 매뉴얼 | 호환 부품 번호 정답 |

### 나라장터와의 조합

```
나라장터 실제 낙찰 데이터 (양식·가격대 참고)
        +
제조사 공식 카탈로그 (스펙 정답)
        ↓
현실적인 가상 견적서 100건 생성 가능
(실제 시장 가격대 반영 + 정답 스펙 기준 오류 삽입)
```

---

## 테스트셋 설계 (Fault Injection)

### 구성

| 유형 | 건수 | 오류 내용 예시 |
| --- | --- | --- |
| **정상** | 50건 | 카탈로그 스펙 일치, 예산 이내, 인증 OK |
| **스펙 불일치** | 20건 | 정격 토크 40Nm → 30Nm로 낮춰 제시 |
| **예산 초과** | 15건 | 단가 상한선 5~15% 초과 |
| **미인증 업체** | 15건 | ISO 9001 미취득 업체명 삽입 |
| **합계** | **100건** | |

### 왜 이 방식이 유효한가

우리가 오류를 직접 심었기 때문에 정답이 명확하다.

```
CWRU 베어링 데이터
  정답: 데이터셋에 부착된 고장 유형 라벨

본 논문 가상 견적서 데이터
  정답: 우리가 의도적으로 심은 오류 유형 라벨
```

두 구조는 동일하다. 이 방식은 정보보안·시스템 연구에서
"Fault Injection Testing"으로 널리 사용되는 검증 방법론이다.

## 하네스 아키텍처

```
[입력: 가상 견적서 (부품번호, 단가, 납기, 업체명, 스펙값)]
        ↓
 ① Input Structuring Layer
    - 비정형 견적서 → 정형 JSON 변환
    - 필수 필드 누락 시 즉시 반환
        ↓
 ② LLM (GPT-3.5급)
    - 카탈로그 스펙 대비 견적서 적합성 판단
    - 추천 여부 + 판단 근거 생성
        ↓
 ③ Validation Loop  ←─────────────────┐
    - 카탈로그 스펙과 수치 불일치 여부 체크  │
    - 예산 상한선 초과 여부 체크          │
    - 인증 요건 충족 여부 체크            │
    - 실패 항목 발생 시 LLM 재질의 ────────┘
        ↓
 ④ Tool Constraint Gate
    - 단가 × 수량 > 승인 한도 → 자동 에스컬레이션
    - 블랙리스트 업체 자동 차단
    - 단일 공급사 의존도 임계치 초과 시 경고
        ↓
 ⑤ Audit Trail 생성
    - 판단 근거·재질의 횟수 전체 로그화
        ↓
[출력: 최종 추천 여부 + 판단 근거]
```

## 논문 구조

### 1. 서론

- 스마트팩토리 부품 조달 오류의 경영적 비용 (라인 스톱)
- 기존 수동 검증의 한계와 LLM 단독 도입의 신뢰도 문제
- 하네스 엔지니어링을 통한 해결 방향 및 Fault Injection 검증 방법론

### 2. 이론적 배경

- LLM의 확률적 특성과 환각(Hallucination) 문제
- 하네스 엔지니어링 개념 정의
  - Validation Loop: LLM 출력을 규칙 기반으로 검증하고 재질의하는 루프
  - Self-Correction: 검증 실패 시 LLM 스스로 출력을 수정하는 메커니즘
  - Tool Constraint: 에이전트의 도구 사용 권한을 사전에 제한하는 구조
- Fault Injection Testing 방법론 개요
- 산업용 로봇 부품 조달 프로세스 구조

### 3. 시스템 설계 (데이터사이언스 파트)

- 하네스 아키텍처 전체 플로우
- 테스트셋 설계: 제조사 카탈로그 기반 Fault Injection 100건
- Validation Rules 정의
  - 스펙 허용 오차 기준 (카탈로그 대비 ±X%)
  - 예산 상한선 설정 기준
  - 인증 요건 체크리스트

### 4. 비용-정확도 분석 (경영 파트)

#### 시나리오 비교

| 시나리오 | API 비용 | 오류 감지 F1 | 실무 투입 가능성 |
| --- | --- | --- | --- |
| 저성능 LLM 단독 | 낮음 | 낮음 | 불가 |
| **저성능 LLM + 하네스** | **낮음** | **높음** | **가능** |
| 고성능 LLM 단독 | 높음 | 높음 | 가능하나 비용 과다 |

- 비용 시뮬레이션: `API 토큰 비용 × 평균 재질의 횟수`
- 오류 미감지로 인한 라인 스톱 기대 손실 절감 추정
- ROI 분석: 하네스 구현 비용 대비 오류 비용 절감

### 5. 논의

- **경영적 시사점:** 구매 전문 인력 없이 1차 스펙 검증 자동화, 공급망 거버넌스 강화
- **한계:** 가상 데이터 기반 검증이므로 실배포 시 현장 데이터로 재검증 필요
- **향후 연구:** 실제 기업 견적 데이터 연동, 멀티에이전트 확장

## 데이터 및 검증 방법

| 항목 | 내용 |

| **검증 방식** | 저성능 단독 / 저성능+하네스 / 고성능 단독 3-way 비교 |
